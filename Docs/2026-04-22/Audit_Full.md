# Audit_Full

## 문서 목적

현재 저장소 상태를 기준으로 프로젝트 운영 구조와 담당 범위(UI 개선, 메일 디자인 개선)를 실행 가능한 형태로 정리한다.

## 1. 프로젝트 목적

| 구분 | 내용 | 상태 |
| --- | --- | --- |
| 프로젝트 성격 | 비상업용, 비수익화, 포트폴리오/학습/공익성 성격의 가벼운 프로젝트 | 확인됨 |
| 서비스 목적 | 주식 지표 조건 충족 시 웹에서 알람을 등록하고 메일로 통지하는 개인용 알림 서비스 | 확인됨 |
| 우선 담당 범위 | 웹 디자인 개선, 메일 디자인 개선 | 확인됨 |
| 비우선 범위 | 알람 조건 고도화는 현재 주 담당 범위가 아님 | 확인됨 |
| 비용 원칙 | 무료 플랜 중심 유지 | 확인됨 |

## 2. 확인된 사실 / 가정 / 미확인 항목

### 2.1 확인된 사실

| 항목 | 내용 | 근거 |
| --- | --- | --- |
| UI 구현 방식 | Streamlit 단일 파일 앱 구조 (`app.py`) | 코드 확인 |
| 메일 발송 방식 | Gmail SMTP + Python `EmailMessage` 기반 | `main.py` 확인 |
| 자동 실행 방식 | GitHub Actions 스케줄러가 `main.py` 실행 | `.github/workflows/alert.yml` 확인 |
| 데이터 저장소 | Supabase 사용 | `app.py`, `main.py`, README 확인 |
| UI 레이아웃 상태 | 기본 Streamlit 컴포넌트 중심, 별도 디자인 시스템/테마 파일 없음 | 코드 확인 |
| 메일 포맷 상태 | 텍스트 메일만 전송, HTML 대체 본문 없음 | `main.py` 확인 |
| 로컬 기본 Python 상태 | 시스템 Python 3.12에는 `streamlit` 미설치 | 실행 확인 |
| 프로젝트 가상환경 상태 | `.venv` 내 Python 3.14.3에서 주요 패키지 import 가능 | 실행 확인 |
| 버전 불일치 | GitHub Actions/Dev Container는 Python 3.11, 로컬 기본 Python은 3.12, `.venv`는 3.14.3 | 실행 및 설정 확인 |
| 문서 불일치 | README는 매시 정각 실행 예시, 실제 워크플로는 10분 간격 | README와 workflow 비교 |

### 2.2 가정

| 항목 | 내용 | 이유 |
| --- | --- | --- |
| Supabase 키 종류 | 현재 앱/배치에서 사용하는 키가 서비스 롤인지 익명 키인지 확정하지 않음 | 코드에는 키 이름만 존재 |
| 테이블 스키마 | `users_pro`, `alerts_pro`가 이미 생성되어 있고 기본 CRUD 권한이 동작 중이라고 가정 | 코드가 직접 조회/삽입 수행 |
| UI 대상 사용자 | 개인 투자자 또는 소규모 테스트 사용자 위주라고 가정 | 프로젝트 목적 및 비용 원칙 기준 |

### 2.3 미확인 항목

| 항목 | 현재 상태 | 문서 반영 방식 |

| 실제 Supabase 테이블 정의 | 저장소 내 스키마 파일 없음 | TODO로 분리 |

| 실제 사용자 피드백 | 확인 자료 없음 | UI 우선순위 판단 시 추가 필요 |
| 브랜드 최종 명칭 | `StockSignalBot Pro`와 `QuantBot` 혼용 | 통일 필요 |

## 4. 현재 UI/UX 문제점

### 4.1 확인된 상태

| 항목 | 현재 상태 | 영향 |
| --- | --- | --- |
| 페이지 구조 | 로그인/회원가입, 알람 등록/관리만 탭으로 구분 | 프로토타입 인상 강함 |
| 시각적 일관성 | 기본 Streamlit 스타일 사용, 공통 색/간격/카드 규칙 없음 | 브랜드/신뢰감 부족 |
| 정보 위계 | 페이지 헤더, 섹션 요약, 도움말, 설명 카드 부족 | 처음 보는 사용자가 맥락 파악 어려움 |
| 입력 경험 | 종목코드 형식 안내, 지표 설명, 성공/실패 후 다음 행동 안내 부족 | 오류 가능성 증가 |
| 관리 화면 | 알람 목록이 expander 중심, 중요 정보 요약 카드 없음 | 스캔 효율 낮음 |
| 빈 상태/오류 상태 | 최소 문구만 존재 | 서비스 완성도 낮아 보임 |

### 4.2 개선 대상

| 수정 대상 | 이유 | 제안 방식 | 기대 효과 |
| --- | --- | --- | --- |
| 첫 화면 헤더 | 서비스 목적과 사용 흐름이 보이지 않음 | 상단에 핵심 가치, 사용 순서, 제한 조건 요약 배치 | 첫인상 개선, 진입 장벽 완화 |
| 로그인/회원가입 영역 | 기능은 있으나 제품 느낌이 약함 | 설명 카드, 입력 힌트, 신뢰 문구, 상태 메시지 규칙 추가 | 프로토타입 인상 감소 |
| 알람 등록 폼 | 입력 항목이 많아도 맥락 제공이 약함 | 지표 분류, 예시값, 조건 설명, 2단 컬럼 재정리 | 폼 이해도 향상 |
| 알람 관리 화면 | 상태 확인이 느림 | 대기/완료 건수를 카드형 요약으로 상단 노출 | 운영 시인성 향상 |
| 시각 언어 | 브랜딩과 정보 위계가 없음 | Streamlit 내 커스텀 CSS 최소 도입, 색/간격/버튼 규칙 정의 | 무료 구조 유지 + 완성도 개선 |
| 피드백 UX | 등록/삭제 후 다음 행동 유도가 약함 | 성공 메시지에 다음 단계 안내 포함 | 사용 흐름 매끄러움 |

### 4.3 권장 구현 방향

| 우선순위 | 권장안 | 비고 |
| --- | --- | --- |
| P1 | Streamlit 유지, `app.py` 내부에 디자인 토큰과 CSS 블록 추가 | 별도 프론트엔드 도입 불필요 |
| P1 | 헤더, 요약 카드, 폼 설명 블록, 알람 카드 뷰 추가 | 현재 담당 범위와 직접 연결 |
| P2 | 지표 설명/예시를 데이터 구조로 분리 | 유지보수성 개선 |
| P2 | UI 변경 전/후 스크린샷 축적 | 이후 PPT/보고자료 전환에 유리 |

## 5. 현재 메일 템플릿/메일 발송 구조 문제점

### 5.1 확인된 상태

| 항목 | 현재 상태 | 영향 |
| --- | --- | --- |
| 메일 형식 | `set_content()` 기반 텍스트 메일만 사용 | 가독성, 신뢰감, 시각적 완성도 부족 |
| 메일 템플릿 위치 | `main.py` 내부 문자열로 직접 조합 | 재사용/수정 어려움 |
| 브랜드 표현 | 제목은 `QuantBot`, 앱은 `StockSignalBot Pro` | 브랜드 혼선 |
| 정보 구성 | 종목코드, 지표, 현재 수치, 조건 정도만 포함 | 전달력 부족 |
| 보조 정보 | 발생 시각, 데이터 기준 시점, 안내 문구, 주의 문구 부족 | 해석 난이도 증가 |

### 5.2 개선 대상

| 수정 대상 | 이유 | 제안 방식 | 기대 효과 |
| --- | --- | --- | --- |
| 메일 템플릿 구조 | 현재 본문이 너무 단순함 | 제목, 요약, 조건, 현재값, 다음 행동, 면책 문구 분리 | 읽기 쉬운 구조 확보 |
| HTML 메일 추가 | 텍스트만으로는 신뢰감 한계 | `EmailMessage.add_alternative()`로 HTML 버전 추가 | 디자인 품질 향상 |
| 텍스트 대체 본문 유지 | 메일 클라이언트 호환성 필요 | plain text + HTML 멀티파트 유지 | 안정성 확보 |
| 브랜드 통일 | 서비스명 혼선 제거 필요 | 단일 브랜드명 확정 후 제목/본문/발신자 일치 | 신뢰감 향상 |
| 템플릿 분리 | 인라인 문자열 수정 비용 큼 | 함수 분리 또는 별도 템플릿 모듈화 | 반복 개선 용이 |

### 5.3 권장 구현 방향

| 우선순위 | 권장안 | 비고 |
| --- | --- | --- |
| P1 | `build_email_payload()` 같은 함수로 제목/본문 생성 분리 | 현재 구조에서 가장 가벼운 개선 |
| P1 | HTML 카드형 메일 1안 제작, plain text 백업 유지 | 무료 환경 유지 가능 |
| P2 | 트리거 시간, 조건 충족 값, 간단 해석 문구 추가 | 정보 전달력 향상 |
| P2 | 샘플 메일 캡처 저장 | 보고자료/PPT 재사용 가능 |
